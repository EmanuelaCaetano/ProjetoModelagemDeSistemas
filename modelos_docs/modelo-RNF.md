# Modelo de Requisitos Não Funcionais

## Instruções

Requisitos não funcionais descrevem **como o sistema deve funcionar**.  
Eles tratam de qualidades e restrições do sistema, como desempenho, segurança, disponibilidade, usabilidade e manutenibilidade.

Use frases claras, objetivas e mensuráveis sempre que possível.

---

## Exemplo de escrita

- O sistema deve responder às consultas em até 2 segundos.
- O sistema deve estar disponível 99% do tempo.
- O sistema deve exigir autenticação para acesso às áreas restritas.

## Categorias comuns de requisitos não funcionais

### Desempenho
Tempo de resposta, volume de acessos, processamento.

**Exemplos:**
- o sistema deve responder em até 2 segundos
- o sistema deve suportar 500 usuários simultâneos

### Segurança
Autenticação, autorização, proteção de dados.

**Exemplos:**
- o sistema deve exigir login
- as senhas devem ser armazenadas de forma criptografada

### Disponibilidade
Tempo em que o sistema deve permanecer acessível.

**Exemplos:**
- o sistema deve estar disponível 99% do tempo
- o sistema deve funcionar 24 horas por dia

### Usabilidade
Facilidade de uso e aprendizado.

**Exemplos:**
- o usuário deve concluir o cadastro com poucos passos
- a navegação deve ser simples e clara

### Confiabilidade
Consistência e correção das operações.

**Exemplos:**
- o sistema deve garantir integridade dos dados
- o sistema não deve perder registros em operações críticas

### Manutenibilidade
Facilidade para corrigir e evoluir o sistema.

**Exemplos:**
- o código deve seguir padrão de organização
- o sistema deve permitir manutenção sem afetar módulos não relacionados

### Portabilidade
Capacidade de funcionar em diferentes ambientes.

**Exemplos:**
- o sistema deve funcionar em diferentes navegadores
- o sistema deve ser acessível em computador e celular

### Escalabilidade
Capacidade de crescer sem perda significativa de qualidade.

**Exemplos:**
- o sistema deve suportar aumento de usuários
- o sistema deve manter desempenho em períodos de pico

---

## Orientações

- escreva requisitos que possam ser verificados
- evite frases vagas como “o sistema deve ser bom” ou “o sistema deve ser rápido”
- prefira requisitos com critério claro, como tempo, volume, percentual ou condição específica
- pense nas qualidades que o sistema precisa ter para atender bem ao problema
